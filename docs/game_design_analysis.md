## 2. 기획(strategy_architecture.md) vs 실제 구현 현황

| 기획된 내용 | 실제 구현 상태 |
|------------|---------------|
| 원소 상태이상 (젖음, 화상, 대전 등) | ❌ 미구현 |
| 원소 연쇄 반응 (감전, 수증기, 대화재 등) | ❌ 미구현 |
| 적의 다음 행동 예고 (Telegraph) | ❌ 미구현 |
| 상인/제단/대장간 노드 (덱 압축 경제) | ❌ 미구현 |
| 캐릭터 고유 패시브 | ❌ descKey만 존재, 로직 없음 |
| 카드 고유 효과 (별 등급별 다른 동작) | ❌ 전부 `key: 'attack'`, 수치만 다름 |
| 히트스탑, 카메라 쉐이크 | ❌ 미구현 |
| 가챠/BM 시스템 | ❌ 미구현 |

**지금 게임은 기획서의 약 20%만 구현된 상태다.**  
나머지 80%가 없는 상태에서 "재미있는가"를 논하는 것은 반쪽짜리 평가다.

## 3. 현재 코드에서 발견한 구체적 문제

### 🔴 문제 1 — 카드 30장 중 25장이 같은 동작

```typescript
// cardData.ts 실제 코드
// 물 1성~5성, 불 1성~5성, ... 모두 동일한 구조
{ key: 'attack', value: 10 }  // 1성
{ key: 'attack', value: 15 }  // 2성
{ key: 'attack', value: 20 }  // 3성
{ key: 'attack', value: 25 }  // 4성
{ key: 'attack', value: 30 }  // 5성
```

25장의 원소 카드가 **"얼마나 세게 때리냐"만 다르다.** 고유 효과가 없다.  
5성 물 카드를 얻었을 때 플레이어의 감정: "강해졌다" (숫자 증가)  
5성 물 카드를 얻었을 때 플레이어가 원하는 감정: "전략이 바뀐다" (가능성 확장)

지금은 전자만 있고 후자가 없다.

---

### 🔴 문제 2 — 배틀에서 실질적 선택이 1가지뿐

배틀 흐름:
1. 5장 드로우
2. **3장 선택** (각 1회 리롤 가능)
3. 공격

"정답"이 항상 같다: **맵 속성에 유리한 카드 중 별이 높은 것 3장.**  
이 규칙을 깨는 이유가 없다.

콤보 보너스가 있지만:
- 같은 카드 3장(+100%): 시작 덱(속성당 5장 × 1성)에서 같은 카드 3장 뽑힐 확률 낮음
- 같은 속성 3장(+40% of median): 의도적으로 노릴 만하지만, 속성 유리/불리 1.5× 배율이 더 강력해서 그냥 유리 속성 3장이 정답

**선택이 전략이 아닌 정렬이다.**

---

### 🔴 문제 3 — 캐릭터 5종이 사실상 1종

```typescript
// CharacterSelectScene.ts 실제 코드
Guardian:  hp:100, atk:10, def:10, crit:10
Ranger:    hp: 60, atk:10, def: 0, crit:20
Berserker: hp: 80, atk:10, def:10, crit:10   // Guardian과 HP만 다름
Titan:     hp:100, atk:10, def: 0, crit:10
Lancer:    hp: 80, atk:10, def: 0, crit:10
```

모든 캐릭터의 **기본 ATK가 동일(10)** 하다.  
Berserker와 Guardian은 HP 20 차이가 전부다.  
`descKey`(고유 특성)는 텍스트만 있고 **배틀 로직에 전혀 반영되지 않는다.**

캐릭터 선택 = "HP 얼마 원하냐"의 선택이다. 플레이 스타일이 달라지지 않는다.

---

### 🟡 문제 4 — 런(Run) 간 차별성이 맵 배치뿐

런마다 달라지는 것: 맵 노드 배치(시드 기반)  
런마다 같은 것:  
- 시작 덱 고정 (1성×5종×5장 = 25장, 항상 동일)
- 캐릭터 스탯 고정  
- 메타 진행 없음 (런 완주해도 다음 런이 더 풍부해지지 않음)

로그라이크의 본질: "이번 런이 왜 지난 런과 다른가"  
현재 답: "맵 배치가 조금 다를 뿐"

---

### 🟡 문제 5 — 적 행동이 불투명하고 단순

```typescript
// BattleScene.ts — 적 행동 로직
// 플레이어 공격 후 → 적 반격 (항상, 조건 없음)
// 다음 턴 적 행동 예고 없음
```

플레이어는 "이번 턴 적이 강공격 예정이니 방어 카드를 써야겠다"는 판단을 할 수 없다.  
방어 카드(defense key)를 쓸 이유가 전략적으로 없다. 적 반격은 어차피 날아온다.

## 4. 긍정적 요소 (살려야 할 것들)

- **노드 다양성**: 10가지 이벤트 타입 (HEART, CARD_FLIP, INDIAN_POKER 등). 트레이드오프가 있는 이벤트(HEART: HP -5 vs 카드밸류 ×1.3)는 **로그라이크 설계의 정석이다.**
- **장비 시스템**: `SpecialEffectType` (shield_on_turn_end, heal_on_win, lifesteal_pct 등) 설계는 Slay the Spire의 유물(Relic)과 비슷한 구조로 **잠재력이 높다.** 이것이 실제로 배틀에 잘 반영되기만 하면 된다.
- **절차적 맵**: 12레이어, 시드 기반 생성은 좋은 구조다.
- **속성 상성**: TYPE_BEATS 시스템은 명확하고 직관적이다.
- **전투 UI**: 데미지 플로팅, 콤보 텍스트, HP바, 쉴드 뱃지 등 피드백은 잘 되어 있다.

## 5. 방향성 제안

### 📌 최우선 (지금 당장) — 전략의 씨앗 심기

#### A. 원소 카드에 별 등급별 고유 효과 추가

`CardData`에 `effectKey`, `effectValue`, `effectDuration` 필드를 추가하면 된다.  
기획서(`strategy_architecture.md`)의 원소 상태이상을 단순화해서 먼저 구현:

```
물 1성: 공격 10 (기존 유지)
물 2성: 공격 15 + [취약] 1턴 부여 (받는 피해 +30%)
물 3성: 공격 20 + [취약] 2턴 부여
물 4성: 공격 25 + 2회 연속 타격
물 5성: 공격 30 + [취약] 3턴, 적 방어력 무시

불 2성: 공격 15 + [화상] (턴 종료 시 5 데미지, 2턴)
번개 2성: 공격 15 + [젖음] 상태이면 감전(×2.5 피해)
```

이것만으로 "어떤 카드를 쓸지"가 전략이 된다. 취약을 먼저 걸고 강한 카드를 쓰는 순서 결정이 생긴다.

#### B. 캐릭터 패시브를 배틀에 실제 적용

`BattleScene.create()` 또는 `executeCardAction()` 내에서 characterId 기반 분기:

```typescript
// 예시
if (characterId === 'berserker' && this.playerCurrentHp < this.data_.playerMaxHp * 0.5) {
  finalDmg *= 1.5; // Berserker 패시브: HP 50% 이하 시 공격력 1.5배
}
```

코드 변경 규모는 작지만 캐릭터마다 플레이 스타일이 완전히 달라진다.

#### C. 적 행동 예고 UI 추가

현재 배틀 화면 적 위에 "다음 행동: 강공격 30" 또는 "다음 행동: 방어" 아이콘을 표시.  
이것만으로 "이번 턴에 방어 카드를 써야 하나"라는 전략적 결정이 생긴다.

---

### 📌 단기 (1~2개월) — 전략 층위 추가

#### D. 상태이상 시스템 (기획서 6.1 간소화 버전)

기획서의 5단계 연쇄 반응은 복잡하다. 먼저 3가지만:
- **취약(Vulnerable)**: 받는 피해 +50%, n턴
- **화상(Burn)**: 턴 종료 시 데미지, n턴
- **방어막(Shield)**: 피해 X 흡수 ← 이미 일부 구현됨

적도 플레이어에게 상태이상을 걸 수 있어야 방어 카드의 가치가 생긴다.

#### E. 덱 구성에 선택권 부여

캐릭터 선택 화면에서 "시작 속성 3개 선택" 또는 "제거할 카드 3장 선택" 추가.  
코드 변경 최소화로 런 초반의 결정이 의미 있어진다.

#### F. 상인/제단 노드 구현 (기획서 6.2)

`NODE_TYPE`에 MERCHANT(상인), ALTAR(제단) 추가 후 `NodeEventScene`에 이벤트 로직 작성.  
카드 제거, 카드 구매, 희생으로 HP 강화 — 이것이 "덱 빌딩"의 핵심이다.

---

### 📌 중기 (3~6개월) — 반복 플레이 이유 만들기

#### G. 메타 진행 (런 간 잠금해제)

런 완주 시 "새 카드 효과 해금", "새 캐릭터 패시브 해금" 등.  
"지금 런은 약해도 미래를 위해 플레이할 이유"가 생긴다.

#### H. 가챠 BM 모델 결정

기획서에 가챠가 언급되지만, 모델이 결정되지 않았다.  
**추천 방향**: 카드 기능은 런 중 공평 획득, 가챠는 "초기 덱 선택 선택권" 또는 "캐릭터 외형".  
핵심 원칙: **실력이 돈보다 중요해야** 장기 플레이어가 생긴다.

## 7. 결론

CB-Tower는 **기반 구조는 올바르다.** 절차적 맵, 노드 다양성, 속성 상성, 장비 시스템은 모두 좋은 방향이다.

**핵심 문제는 하나다:**

> 카드에 고유 효과가 없으면, 카드 게임이 아니라 숫자 게임이 된다.

`strategy_architecture.md`에 이미 좋은 원소 반응 기획이 있다.  
이것을 **단순화해서 일단 작동하게** 만드는 것이 최우선이다.

카드에 효과가 생기면 → 콤보와 순서 결정이 의미 있어진다  
콤보가 의미 있어지면 → 덱 구성(어떤 속성/별 을 모을지)이 전략이 된다  
덱 구성이 전략이 되면 → 노드 이벤트의 선택(CARD_SWAP으로 무엇을 바꿀지)이 의미 있어진다  
노드 이벤트가 의미 있어지면 → 맵 탐색(어느 길로 갈지)이 진짜 결정이 된다

**지금은 이 연쇄가 첫 고리부터 끊어져 있다. 카드 효과부터 시작해라.**
